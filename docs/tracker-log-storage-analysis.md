# Tracker Log Storage Analysis

Sizing and cost analysis for the `tracker_log/log.json` archival feature, plus
options for moving the data off-device to cloud storage.

## Empirical baseline

Measured from a real `tracker_log/log.json` file generated by the current
implementation in Chicago (60657, 70 nm radius).

| Metric | Value |
|---|---|
| File size | 15.22 MB |
| Time span covered | 3.06 hours |
| Unique callsigns | 1,349 |
| Total entries | 24,225 |
| Avg entries per callsign | 18 |
| Avg bytes per entry | 659 bytes |
| Entry rate | ~7,910 entries/hour |
| Growth rate | ~4.97 MB/hour |
| Flights per poll | ~92 |
| Poll interval (measured) | 41.9 s (matches the 42 s code path) |

Sample was taken during peak daytime hours; weekly average is roughly 60–70% of
this rate due to diurnal traffic variation (overnight drops to ~20–30% of peak).

## Local-storage projections

Steady-state file size for the existing JSON-on-disk approach:

| TTL window | Realistic size | Peak-rate size | Entries | RAM (parsed) |
|---|---|---|---|---|
| 168 hours (1 week) | ~600 MB | ~835 MB | ~1.3 M | 1.5–2.5 GB |
| 84 hours (3.5 days) | ~300 MB | ~420 MB | ~660 K | 750 MB – 1.25 GB |
| 24 hours | ~80 MB | ~120 MB | ~190 K | 200–400 MB |

The Pi has 4 GB RAM, so even the 168hr window is technically viable, but RAM
usage runs 40–60% of total.

### Operational concerns at scale

Issues that show up before the disk size itself becomes a problem:

1. **`save_file` blocking** — synchronous `json.dump` of the whole dict every
   2 minutes. At 600 MB this could pause the worker thread for several seconds
   and potentially miss a poll cycle.
2. **`cleanse_log` cost** — walks every entry on every `update_log` call
   (every 42 s). At ~1.3 M entries that's real CPU on every poll.
3. **Cold-start parse time** — `_load_file` parses the full JSON on startup;
   multi-second freeze at 600+ MB.

Cheap mitigations if staying on local storage:
- Cleanse only the callsign that was just updated, not the entire dict
- Move `save_file` off the polling thread (background writer)
- Consider switching to a streaming append format rather than rewriting the
  whole file every save

## Cloud storage options

Pricing reflects `us-east-2` (Ohio, matches `AWS_REGION` in `config.py`).
All estimates use realistic diurnally-adjusted volume:
~2.4 GB/month ingestion, ~300–600 MB steady-state storage.

### Summary

| Option | Cost/month | Best for | Weakness |
|---|---|---|---|
| **S3** (batched JSON) | ~$0.06 | Cheapest archive, Athena-queryable | Awkward for live access |
| **DynamoDB** (provisioned) | ~$1.00–1.40 | Query by callsign, real TTL, structured | Most ops overhead |
| **CloudWatch Logs** | ~$1.20 | Live tail, alarms, easy setup | Logs-style, not records |
| **DynamoDB** (on-demand) | ~$4.60 | No capacity planning | Most expensive |

**AWS Free Tier (first 12 months on a new account)** covers all of these
workloads entirely — you'd pay $0 for the first year.

### CloudWatch Logs

- **Ingestion**: $0.50/GB → ~$1.20/month at 2.4 GB
- **Storage**: $0.03/GB/month on compressed size (~10×) → negligible
- **Logs Insights queries**: $0.005/GB scanned → negligible unless very frequent
- Plus ~26 bytes overhead per event

Best for: live tail-style monitoring, simple text search via Logs Insights,
setting up alarms on traffic patterns.

### DynamoDB

Each ADSB entry (~659 bytes) consumes 1 WRU (1 KB cap).

| Cost component | Amount |
|---|---|
| On-demand writes ($1.25/M WRU) | ~$4.60/month |
| Provisioned writes (3 WCU at peak) | ~$1.40/month |
| Provisioned with auto-scaling (2 WCU avg) | ~$1.00/month |
| Storage ($0.25/GB after 25 GB free) | $0 (under free tier) |
| TTL-driven deletes | $0 (free feature) |
| Reads (see below) | Near $0 for any normal usage |

**Provisioned > on-demand** for this workload because the rate is predictable
(~1.4 writes/sec average, ~2.2/sec peak). On-demand only wins if traffic is
spiky or unpredictable.

#### DynamoDB TTL

DynamoDB has built-in TTL designed exactly for this use case:

1. Designate one attribute as the TTL attribute (e.g., `expires_at`)
2. Value must be **Unix epoch seconds** — note current `timestamp` field is in
   milliseconds; divide by 1000
3. DynamoDB auto-deletes expired items in the background
4. **Deletes consume no WRUs** — free
5. Replaces `cleanse_log` entirely

Setup (one-time):

```bash
aws dynamodb update-time-to-live \
  --table-name tracker_log \
  --time-to-live-specification "Enabled=true, AttributeName=expires_at"
```

On insert:

```python
expires_at = int(time.time()) + (168 * 3600)  # 168 hours from now
```

**Gotchas:**
- Deletion is best-effort; the SLA is "typically within 48 hours of
  expiration." Items may remain readable past their TTL.
- Expired-but-not-yet-deleted items still count toward storage and are still
  returned by queries. Filter on `expires_at > now()` client-side if exact
  cutoffs matter.
- TTL is per-item, so different items can have different windows.
- Deletions can flow through DynamoDB Streams if you want to archive to S3
  before they're gone.

#### Read costs for ad-hoc lookups

For querying from a local tool (e.g., "show me every time UAL1234 passed
overhead"):

- 1 RRU = 1 strongly consistent read up to 4 KB (or 0.5 RRU eventually
  consistent)
- One flight's worth of data (~50–200 entries, ~130 KB) ≈ 33 RRUs
- Cost per lookup: **~$0.0000083** (strongly consistent)

| Lookup volume | Cost |
|---|---|
| 1 lookup | ~$0.000008 |
| 1,000 lookups | ~$0.01 |
| 100,000 lookups | ~$0.83 |

API calls themselves are not billed — only consumed RRUs/WRUs are. The first
100 GB of data transfer out to the internet per month is free.

#### Design rule

Use `callsign` as **partition key**, `timestamp` as **sort key**. Then a flight
lookup is a single `Query` operation — instant and effectively free. Avoid
`Scan` (full table read) — a 300 MB table scan consumes ~75,000 RRUs (~$0.02
per scan), still cheap but ~2,000× more expensive than a targeted query.

### S3 (batched JSON)

Cheapest archival option:

- ~$0.023/GB-month for Standard storage
- ~2.4 GB/month → ~$0.06/month
- Queryable via Athena ($5/TB scanned)

Best for: durable archive when you don't need live access. Less convenient for
individual record access; requires batching logic in the tracker.

## Recommendation

If the goal is just to keep historical data around for occasional lookup
("did this plane fly over last Tuesday?"), **DynamoDB with provisioned capacity
and TTL** is the sweet spot:

- ~$1/month after free tier (free for the first year)
- Built-in retention handling — no `cleanse_log` walking
- Designed for the access pattern (`Query` by callsign)
- Eliminates the RAM/cold-start/save-blocking concerns of the local file
- Pi only needs to write; reads happen from the local lookup tool

If the goal is real-time monitoring of *the tracker itself* (logs, alerts),
CloudWatch Logs is a better fit, but that's a separate concern from the
archival feature.
